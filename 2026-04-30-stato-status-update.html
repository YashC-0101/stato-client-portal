Faster What got cached today

Yesterday: 4 windows of /reports/campaign stayed warm. Today: every LeadByte read path the dashboard touches stays warm — independent of which time-window selector you click.

1 · All 7 time windows of /reports/campaign now prewarmed

Yesterday the LeadByte Dashboard window selector had four cached windows: today, this_week, this_month, ytd. If you clicked yesterday, last_week, or last_month — those hit LeadByte live (~1.5s wait).

Today the prewarmer extends to all 7 windows. Click any window in the selector → ~50 ms response. Backend code reads through the same cache key as the prewarmer writes, so first-user-of-the-day still gets a warm hit.

Test it: Sidebar → LeadByte → Dashboard → click through Today / Yesterday / This Week / Last Week / This Month / Last Month / Year to Date — every switch should be near-instant.

2 · Supplier Spend per window — new endpoint + cache

New backend endpoint /api/v1/leadbyte/reports/supplier-spend?window=… cached per window (5-min TTL, prewarmed every 90 s alongside campaign reports).

Drives the Supplier Spend table on the LeadByte Dashboard page — shows which suppliers brought in leads + how much was spent per supplier for the selected window.

Test it: Sidebar → LeadByte → Dashboard → scroll to the Supplier Spend section → switch window selector — table refreshes with cached data each time.

3 · Aggregated dashboard summary — new endpoint

New endpoint /api/v1/leadbyte/reports/summary?window=… — returns the totals you see at the top of the LeadByte Dashboard (LEADS / VALID / REVENUE / PAYOUT / PROFIT) in a single API call.

Reuses the same Redis cache key as /reports/campaign, so the summary tiles + the campaign table + the supplier spend table all load from one warm cache entry — no triple-hitting of LeadByte.

Why it matters: the dashboard window selector previously loaded 3 endpoints in parallel; now they share work, fewer network calls, faster first paint.

4 · Buyers + Deliveries listings now cached

The full LeadByte Buyers list (52 entries) and Deliveries list now cache to Redis on both the prewarmer and the read path. Filtered queries (status / bid filters) bypass the cache so search results stay live; the unfiltered listing — the common case for the index pages — loads from cache.

Test it: Sidebar → LeadByte → Buyers — list should appear instantly even on a cold tab. Same for LeadByte → Deliveries.

5 · LeadByte sync timestamp — now updates every 90 s

Settings → Integrations → LeadByte tile shows a Last sync timestamp. Yesterday it only updated when the 2-min sync job ran; today the prewarmer also marks a sync timestamp on every successful cycle, so the tile reads "Last sync: a few seconds ago" any time the platform is healthy.

Side benefit: if Sam ever opens Settings and the tile shows a stale timestamp (say, 10+ minutes), that's an immediate signal something is wrong — the worker isn't refreshing the cache.

Polish Smaller UI tweaks

Client detail — only show LeadByte ID when actually linked

On the Clients → [client] → Overview tab → "External System IDs" card, the LeadByte Client ID row only appears when the client has one. Previously it always rendered "Not linked" — misleading for the typical leadgeneration.io client where linkage runs at the campaign level rather than the client level.

Endole + Xero IDs still show "Not linked" when absent, since those are expected to populate (Xero auto-creates on agreement signing; Endole on credit-check trigger).
