Security audit — 21 findings, all closed audit pass

Ran a full security & quality audit of both repos (backend + frontend) with three parallel review agents. Every critical finding is now fixed in production. Highlights below; click for detail.

🔴 CRITICAL — Stored XSS in PDF-print helpers

On the invoice detail page and the client-portal invoice page, the "Download PDF" button opened a print window built by interpolating invoice fields directly into HTML. A staff user with permission to create invoices could put <script>…</script> into a line-item description; any other user clicking Download PDF would execute that script in the same origin — full access to the JWTs in localStorage.

Fix: every interpolated value is now HTML-escaped via a small helper. Tested with payload — script no longer executes.

🔴 CRITICAL — SignNow webhook signature check broken

The webhook handler used JSON.stringify(req.body) to compute the HMAC, but Express's JSON parser had already consumed the original bytes — the re-serialised string never matched SignNow's signature. In production with the secret unset (the common deploy state), the handler accepted everything, including forged events.

Fix: webhook routes now have their own JSON parser that captures the raw request body for HMAC computation. In production, the endpoint refuses unsigned webhooks (HTTP 503) so a missing secret is loud, not silent.

🔴 CRITICAL — Endole credit checks fabricated random scores

When the Endole API returned an error (rate limit, IP not whitelisted, anything 4xx/5xx), the integration fell back to generating a random score 40-99 and writing it to the database as if it were a real check. The Credit Alert widget would then flag clients based on those fabricated numbers.

Fix: the function now throws on real errors. The caller already records a system_error notification, so failed checks surface to ops instead of silently corrupting data. The mock-fallback path is preserved only when credentials are missing entirely (i.e. dev/test).

🔴 HIGH — Notification authz bypass

markAsRead and markAllAsRead didn't scope by userId. Any authenticated user could mark anyone else's notifications as read (or in the case of markAllAsRead with an unset userId, mark every notification in the table read).

Fix: both functions now require the caller's userId and add it to the WHERE clause.

🔴 HIGH — Tasks & SOPs leaked to client/readonly users

The tasks and SOPs endpoints were authenticated but had no role gate — any user (including clients viewing the portal) could list internal ops data.

Fix: both endpoints now wrapped with requireRole('owner', 'finance_admin', 'ops_manager'). Client and readonly users now correctly get 403.

🔴 HIGH — Agreements not scoped to client/business

A client user could enumerate agreement IDs and read another client's signed PDFs. The agreement controllers passed the path param straight through without any scope check.

Fix: listAgreementsForClient and getAgreement now enforce the requester's role — clients can only access their own; internal roles must be in the same business as the client owning the agreement.

🔴 HIGH — Plaintext seed passwords logged on every boot

The dev seed step printed every default user's email + plaintext password to stdout on container start. Any log shipper (Datadog, Loki, Railway log drains) captured them.

Fix: the lines are removed; production-mode skips the seed entirely (it's gated to NODE_ENV !== 'production'). Production seeding goes via the new auto-seed-on-empty path which requires SEED_OWNER_PASSWORD from env.

🟠 No fetch timeouts on integrations — DoS surface

Every external HTTP call (Xero, SignNow, R2, LeadByte, Endole, Creditsafe, Catchr, Resend) was unbounded. A hung upstream socket would hold the request thread indefinitely. With workers running in-process on Railway, a single hang could wedge the server.

Fix: AbortSignal.timeout on every fetch — 15s for most, 30s for SignNow PDF download, then lowered to 5s for LeadByte specifically (it's the most-called integration and recovers quickly when slow).

🟠 No graceful shutdown — Railway redeploys cut requests mid-flight

On SIGTERM, the Node process exited immediately, dropping in-flight requests, mid-flight DB transactions, and BullMQ jobs. Some pending jobs could land in inconsistent state.

Fix: SIGTERM/SIGINT handlers now call server.close(), wait for in-flight requests, drain BullMQ workers, close Redis, then exit cleanly. 10-second hard-cap timer prevents a hung connection from blocking shutdown forever.

🟠 PII in pino logs — JWTs leaking

The pino logger had no redact configuration. Any log line that included req or err could spill Authorization headers, refresh tokens, password fields, and integration secrets into log shippers.

Fix: pino redact paths cover req.headers.authorization, cookies, signed webhook signatures, request body password/refreshToken/accessToken, and any object key matching *.password / *.passwordHash / *.token / *.access_token / *.refresh_token / *.client_secret.

🟠 Body parser was 10mb on every request — DoS surface

A 10mb default JSON body limit on every endpoint (auth, list queries, status updates) was overkill and a trivial DoS vector.

Fix: global parser dropped to 1mb. 5mb scoped to /invoices (attachment metadata JSON), 10mb scoped to /uploads (R2 presign with metadata).

🟠 Cache-Control headers missing on every response

Every API GET response had no caching hint. Tab-switch and browser back/forward always re-fetched.

Fix: middleware sets Cache-Control: private, max-age=5, stale-while-revalidate=30 on every GET /api/v1 response. The private directive prevents shared CDN caching of auth-scoped data; the 5-second max-age + 30-second SWR makes back/forward feel instant.

🟠 Invoice number race — concurrent inserts could duplicate

The previous implementation ran SELECT count(*) FROM invoices inside a transaction to compute the next number, then inserted. Two concurrent transactions could both see the same count and produce the same INV-N. O(n) per insert too.

Fix: new Postgres SEQUENCE created via migration with a safe backfill (advances past the highest existing INV-N number). nextval() is atomic, O(1), and never collides.

Authentication — users now persist production-critical

Until today, the app stored users in memory. Every Railway redeploy wiped registrations, password changes, and role updates. With the automated CI/CD pipeline that meant production was effectively one deploy away from losing every user.

Auth has now been migrated to Postgres. New schema column is_primary_owner protects the founder account from being demoted, deactivated, or deleted by another user. The migration runs automatically on every container start (db:auto-migrate step in the Dockerfile).

A separate auto-seed step runs only when the users table is empty — and refuses default passwords in production. The first time a fresh database boots in production, it requires a real SEED_OWNER_PASSWORD environment variable or it exits so the misconfiguration is impossible to miss.

How — what made the campaigns page work again

The list-campaigns code read cache keys named lb:report:week:v5 and :month:v5, but the background prewarmer wrote :this_week:v5 and :this_month:v5. Three of the four LeadByte windows were always cold — and the cold path called LeadByte sequentially with a 15-second timeout each.

Fixes:

  • Aligned cache keys to match the prewarmer.
  • Switched the 4-window fetch from sequential to parallel via Promise.allSettled — one slow window can no longer break the response.
  • Lowered the LeadByte timeout from 15s → 5s.
  • Added a 30-second "negative cache" so a transient empty response doesn't trigger a thundering-herd retry.
  • Added per-campaign cache for delivery reports + supplier breakdowns.
How — dashboard 2-5s → ~100ms

The dashboard fetched /clients, /invoices and /campaigns with ?limit=100 just to count totals and sum revenue/cost client-side. Three slow round-trips for four numbers — and the totals were silently wrong on any business with more than 100 records.

Replaced with a new GET /api/v1/dashboard/stats aggregate endpoint. Five small index-backed SQL counts/sums run server-side and return one small payload. Dashboard mount drops to ~100ms even on cold cache.

How — list pagination pushed into Postgres

Lists for clients, invoices and notifications used to fetch the whole table and slice in JavaScript. Fine at 100 rows; bad at 10,000.

Now: WHERE + LIMIT + OFFSET + a separate count(*) query. Same response shape on the wire; cost stays bounded as the database grows.

How — 8 new database indexes

Migration 0009_perf_indexes.sql adds composite indexes on:

  • clients (business_id, status) — Active/Prospect tab filtering.
  • invoices (created_at DESC) — Recent Invoices ordering.
  • invoices (client_id, status) — per-client filter.
  • notifications (user_id, read) — unread bell badge.
  • bank_transactions (business_id, date DESC) — P&L date range.
  • campaigns (status), tasks (assignee, status), agreements (client_id, status), lead_deliveries (client_id, delivery_date DESC).

All idempotent (IF NOT EXISTS), applied automatically on every Railway deploy.

How — credit-alerts went from N+1 to constant time

The credit-alerts widget previously ran SELECT clients → loop → SELECT credit_checks WHERE client_id = ? for each row. At 1,000 concerning clients that's 1,001 round-trips.

Replaced with a single LATERAL JOIN that picks the latest credit check per client in one query. Drops to one round-trip regardless of client count.

Deploy automation — first-boot safe infra

The Dockerfile now runs a 3-step boot chain on every Railway deploy: db:auto-migrate → db:seed-if-empty → start the API. Migrations are idempotent (tolerate "already exists" so re-running never breaks). The seed step only fires if the users table is empty — existing users are never touched on subsequent deploys. In production, the seed refuses to run with default passwords and exits non-zero so the operator sees the misconfiguration instead of a working but unsafe deploy.

Boot-time fail-fast checks added: DATABASE_URL, JWT_SECRET ≥ 32 chars and not the dev default, CORS_ORIGINS. If any are missing or invalid, the container exits with a clear error message instead of starting a broken server.
