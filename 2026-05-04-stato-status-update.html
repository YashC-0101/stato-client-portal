How — what made the campaigns page work again

The list-campaigns code read cache keys named lb:report:week:v5 and :month:v5, but the background prewarmer wrote :this_week:v5 and :this_month:v5. Three of the four LeadByte windows were always cold — and the cold path called LeadByte sequentially with a 15-second timeout each. Under brief LeadByte slowness this triggered a 60-second wall and a thrown 500.

Fixes:

  • Aligned cache keys to match the prewarmer.
  • Switched the 4-window fetch from sequential to parallel via Promise.allSettled — one slow window can no longer break the response.
  • Lowered the LeadByte timeout from 15s → 5s.
  • Added a 30-second "negative cache" so a transient empty response doesn't trigger a thundering-herd retry.
  • Added per-campaign cache for delivery reports + supplier breakdowns.
How — dashboard 2-5s → ~100ms

The dashboard fetched /clients, /invoices and /campaigns with ?limit=100 just to count totals and sum revenue/cost client-side. Three slow round-trips for four numbers — and the totals were silently wrong on any business with more than 100 records.

Replaced with a new GET /api/v1/dashboard/stats aggregate endpoint. Five small index-backed SQL counts/sums run server-side and return one small payload. Dashboard mount drops to ~100ms even on cold cache.

How — list pagination pushed into Postgres

Lists for clients, invoices and notifications used to fetch the whole table and slice in JavaScript. Fine at 100 rows; bad at 10,000.

Now: WHERE + LIMIT + OFFSET + a separate count(*) query. Same response shape on the wire; cost stays bounded as the database grows.

How — 8 new database indexes

Migration 0009_perf_indexes.sql adds composite indexes on:

  • clients (business_id, status) — Active/Prospect tab filtering.
  • invoices (created_at DESC) — Recent Invoices ordering.
  • invoices (client_id, status) — per-client filter.
  • notifications (user_id, read) — unread bell badge.
  • bank_transactions (business_id, date DESC) — P&L date range.
  • campaigns (status), tasks (assignee, status), agreements (client_id, status), lead_deliveries (client_id, delivery_date DESC).

All idempotent (IF NOT EXISTS), applied automatically on every Railway deploy.

How — credit-alerts went from N+1 to constant time

The credit-alerts widget previously ran SELECT clients → loop → SELECT credit_checks WHERE client_id = ? for each row. At 1,000 concerning clients that's 1,001 round-trips.

Replaced with a single LATERAL JOIN that picks the latest credit check per client in one query. Drops to one round-trip regardless of client count.

Authentication — users now persist production-critical

Until today, the app stored users in memory. Every Railway redeploy wiped registrations, password changes, and role updates. With the new automated CI/CD pipeline that means production was effectively one-deploy-away from losing every user.

Auth has now been migrated to Postgres. The migration runs automatically on every container start. A new automatic-seed step creates the initial owner user only when the database is empty — and refuses to use default passwords in production. The first time a fresh database boots in production, it requires a real SEED_OWNER_PASSWORD environment variable or it exits so the misconfiguration is impossible to miss.

Security — audit findings closed hardened

  • Stored XSS in invoice + portal print-PDF helpers — every interpolated value now HTML-escaped. A malicious line-item description can no longer execute scripts in the print window.
  • SignNow webhook signature verification fixed — was using JSON.stringify(req.body) (which never matches the original bytes); now captures raw body. Production refuses unsigned webhooks if the secret is missing.
  • Endole no longer fabricates credit scores on errors — was returning random mock scores when the API failed. Now throws cleanly so the system records a notification instead.
  • Notification scope check — mark-as-read now requires the calling user own the notification. Previously a stale authz path let users mark anyone's notifications.
  • Tasks + SOPs locked to internal roles — clients and read-only users now get 403 (was 200, exposing internal ops data).
  • Agreements scoped per-business — clients can no longer guess UUIDs to read another client's signed PDFs.
  • 15-second AbortSignal on every external integration — Xero, SignNow, R2, LeadByte, Endole, Creditsafe, Catchr, Resend. No more hung sockets stalling the worker.
  • Graceful shutdown on SIGTERM — drains in-flight requests, closes Redis, then exits. Prevents corrupted writes during Railway redeploys.
  • PII redaction in logs — JWTs, refresh tokens, passwords, integration secrets — all redacted before anything reaches Datadog/Loki/Railway log drains.
  • Cache-Control headers on every API GET response — browser back/forward feels instant; CDN-private so auth-scoped data never leaks across users.
  • Body-parser limits tightened — global default dropped from 10mb to 1mb; 5mb scoped to invoices, 10mb scoped to uploads. Reduces DoS surface on the bulk of requests.
  • Invoice number race fixed — was using count(*) inside a transaction; now a Postgres SEQUENCE via nextval(). Atomic, O(1), no concurrent-write duplicates.
