One client end-to-end live

Your Loom asked: "let's just get one client set up, fully processed through." Every complaint from that walkthrough is addressed below. Tested end-to-end with an 11-step automated test on UK Energy Saving Network.

✅ Address split into 5 fields

Your ask: "I would prefer to have the addresses like address, towns, county, country, postcode."

Single text field replaced with five: Address Line, Town, County, Country (defaults to United Kingdom), Postcode. Existing client records keep their old free-text address until updated.

✅ Multiple contacts per client

Your ask: "we should have multiple contacts in here because you might have a billing contact, you might have a compliance contact or primary contact."

The Contact Details card is now a repeater. First row is locked as Primary; click "Add billing contact" / "Add compliance contact" / "Add other" to add more. Each row has Name / Role / Email / Phone plus a coloured type badge on the detail page (primary blue, billing green, compliance purple, other grey).

✅ VAT number + rate on the client

Your ask: "client VAT number, VAT applicable, like we need to have the VAT number to be able to put that in."

When the "VAT Registered" checkbox is ticked, two new inputs appear: VAT Number (e.g. GB123456789) and VAT Rate % (defaults to 20). Both persist on the client and surface on the Billing card of the detail page.

✅ LeadByte buyer is now a dropdown

Your ask: "We should be able to just select it from a dropdown list from Lead Byte rather than having to go into Lead Byte and do it ourselves."

Replaced with a dropdown fed from /api/v1/leadbyte/buyers. Options labelled {company} · {bid}, sorted by company. Graceful fallback to a text input if LeadByte is unreachable — so a LeadByte outage doesn't block client creation.

✅ Documents now in the database, not browser localStorage

Your ask: "we've got a couple of our agreements, we've got a due diligence form we've completed." Previously these were only visible on the browser that uploaded them.

New client_documents table tracks metadata (uploader, size, content-type, R2 key) centrally. Files still live in Cloudflare R2. Upload from one browser → log in from any other browser → docs are there. Survives cache clears, survives staff turnover.

✅ Invoices tab pulls real Xero invoices

Your ask: "there's no invoices showing from Xero. It's not connected. This isn't working."

Tab now displays a proper invoices table for this specific client: invoice number, status badge, due date, amount, days-overdue chip. Total revenue shown in the header.

✅ "Sync from Xero" button — auto-links the Xero contact by name

Your ask: "Where's his invoice? How do I connect his invoices?"

New button on the Invoices tab. On click:

  • Looks up the client's Xero contact ID. If missing, searches Xero for an exact name match and saves the link automatically.
  • Pulls all ACCREC (customer) invoices for that contact.
  • Imports any not already in our DB; skips duplicates by Xero invoice ID.
  • Shows a toast: "Imported N invoices · auto-linked contact" / "Already up to date" / "Couldn't find …, create contact in Xero first".

Idempotent — running it again just confirms nothing new arrived.

✅ "Validation failed, billing workflow invalid" — gone

Your ask: "Validation failed, billing workflow invalid. So I had to put weekly on there, which is not what I wanted to do."

Root cause was a backend / database enum mismatch: ['weekly_auto', 'monthly_auto', 'manual'] in the validation layer vs ['weekly_auto', 'monthly_validated', 'custom'] in the database. Your frontend dropdown was sending the correct database values which the validation then rejected. Aligned both sides — the toast doesn't fire any more.

Campaigns inverted — Solar Panels is one row, buyers underneath live

Your Loom: "A campaign is solar panels as a campaign, but under that we have multiple clients and multiple deliveries. I don't get this campaign thing because it's a delivery, not a campaign." The data model now matches that mental model.

✅ Campaign is now a vertical, not a (vertical × client) pair

Old model: one row per (client × vertical). "Solar Panels" showed up N times if N clients bought leads on it. Confusing because you'd see three "Solar Panels" entries that were really the same thing.

New model: campaign = the vertical (Solar Panels). Buyers link underneath via a new client_campaigns table with per-buyer lead price + status. Migration is non-destructive — every existing campaign row was backfilled into the new join so nothing was lost.

Note: legacy rows from the old model still exist in the database for back-compat. You may still see multiple "Solar Panels" entries in the grouped view (one per pre-existing buyer link) until those legacy rows are manually deduped. The new ONE-row behaviour applies to vertical-only campaigns created from now on.

✅ Cost-per-lead field, editable inline

Your ask: "You've got no cost per lead."

New "Cost per lead" card on the campaign detail page. Pencil icon opens an inline editor; Save persists to the database. Distinct from the LeadByte-computed CPL (which is totalCost / totalLeads) — this one is your supplier-cost target, showed alongside.

✅ Traffic-source mapping table — leadreports.io shape

Your ask: "we selected Facebook, we selected the supplier, you know, it's shown a supplier, we've connected the NCP, ad spend's there, bam, we've got net profit, we've got five leads."

New table on the campaign detail page:

  • Columns: Name / Supplier / Catchr NCP / Spend / Leads / CPL / Revenue / Profit / Margin
  • "Add source" button opens an inline row with a supplier dropdown (Facebook / Google / Bing / TikTok / Taboola / Outbrain / Other) and Catchr NCP URL input
  • Per-row edit (pencil) and delete (trash)
  • Revenue = campaign.leadPrice × supplier-leads. Profit and margin compute server-side and surface coloured (green/amber/red) based on margin band
✅ Campaigns list grouped by vertical (default on)

New "Group by vertical / Flat" toggle on the campaigns list page. Default is grouped. "Solar Panels" appears as one collapsible header with its buyer count, aggregate month-leads, total revenue, and margin — click the chevron to expand and see each buyer underneath. Toggle to Flat for the old per-row table. Preference persists per staff member.

Five separate report pages (Campaign Performance, Supplier Performance, Ad Spend, Client P&L, Financial Overview) are deleted. /reports now lands directly on the unified view. Old URLs (/reports/campaign, /reports/supplier, /reports/ad-spend, /reports/client-pnl, /reports/financial) all redirect — your bookmarks still work.

Dashboard widgets — Sam-Loom-driven corrections live

✅ Bank widget now uses statement balance (not Xero ledger)

Your Loom: "the actual bank balance, statement balance on May 7th was 52,446 pounds. But we're showing this balance in Xero, which is showing us 113,515."

The widget previously read the BankSummary report's closing balance — that's the ledger value, which includes approved-but-unreconciled transactions and so doesn't match the bank's statement balance.

Now reads Account.Balance from Xero's /Accounts endpoint, which is the bank-feed-reported figure that matches what you see on Xero's own dashboard widget. Description updated to "Statement balance · live from Xero" so it's explicit. If you still see £113k after this update, bank-feed config in Xero may need re-syncing — flag it and I'll dig deeper.

✅ VAT widget shows past AND current quarter

Your Loom: "VAT liability since end of quarter zero, which is not correct... I need to be able to see what my VAT liability is for the 31st, for the February 1st, February, March, April... today my VAT due is £1,261."

Previously showed one accruing period since the last HMRC-default quarter-end (Mar/Jun/Sep/Dec). Two issues:

  • Clinical Marketing Solutions's actual VAT registration uses quarter-ends on Apr/Jul/Oct/Jan — not the HMRC default. Hard-coded the right quarters into the backend.
  • You wanted to see two numbers, not one. Now the widget shows both: past quarter due (e.g. £38,484 for Feb-Apr) emphasised large at the top, current quarter accruing (e.g. £1,261 for the running May-Jul period) below it. Each row labelled with the month range.
