7 · Demo client flows fixed — Send Agreement / Create Invoice / Upload Creative

Three flows were silently broken when run against the Apex Media Ltd (for demo) client. Validation rejected the demo seed UUID (Zod 4's strict .uuid() doesn't accept UUIDs whose version nibble is 0, even though Postgres accepts them and they're valid identifiers).

Replaced with a shared lax UUID-shape check that mirrors what Postgres allows. Now Send Agreement, Create Invoice, and Upload Creative all work against the demo client (and any other client). Added uuid-shape.test.ts guarding all three endpoints against future regressions.

Test it: Clients → "Apex Media Ltd (for demo)" → click Create Agreement in header → upload a PDF → Send. Should succeed (was failing with "Invalid input" before).

8 · Validation errors now show the actual failing field

Helps you (and Sam during testing) actually see what to fix instead of guessing.

14 · Auth security — verify password before disclosing account state

Login was returning "Account is disabled" before validating the password, so a wrong-password attempt on a disabled account would leak the fact that account exists + is disabled. Reversed the order: password is verified first; we only return the disabled-state message after the caller has proven they hold the correct credentials.

Defensive fix — no real-world impact (no disabled accounts in production yet) but good hygiene before that comes up.

15 · Rate limit raised — "Too many requests" no longer fires mid-session

From your 30-Apr test report: Bank Feed → Apply category dialog showed "Too many requests, please try again later" mid-flow. The general API cap was 100 requests per 15 minutes — too tight for a normal admin session (dashboard polls + LeadByte 90 s refetches + pagination + categorisation clicks easily clear 100 in a few minutes).

Raised general cap to 1500 / 15 min (~100 rpm). Login limiter stays tight at 20 / 15 min against brute-force. Limits are exported as constants and pinned with tests so a future tightening can't silently regress this flow.

Test it: Bank Feed → click rapidly through transactions + Apply category — no 429 errors.

17 · Sidebar — icons centered in collapsed mode

When the sidebar is collapsed (icon-only), nav items now centre their icons in the column instead of left-aligning. Small visual fix; makes the collapsed sidebar feel intentional rather than truncated.

18 · LeadByte → Buyers / Deliveries pages now load (was hanging "loading" forever)

From your test: "Sidebar → LeadByte → Buyers — the data did not load after 1 min", same on Deliveries.

Root cause: LeadByte's list endpoints return {status: 'Success', message: 'OK', buyers: [...]} (envelope), not a bare array. Backend's getBuyers() was blindly casting the envelope as LeadByteBuyer[] — so the frontend received an object, tried to .map() over it, and got stuck in a render loop with no error surfaced. Same bug hit Deliveries and Responders.

Fix: new unwrapList<T>(res, primaryKey) helper that digs into the named array key (or data, or treats as a bare array if already so). Applied to all three list functions. 4 regression tests pin the unwrapping including the empty-envelope safety case.

Test it: Sidebar → LeadByte → Buyers should now show 52 buyers (The Solar Geeks, Amplifon CH, …). Sidebar → Deliveries shows 94 delivery routes.

Faster Performance overhaul

Dashboard load: 3-5 seconds → ~500ms (5-10× faster)

Three layered fixes:

  • N+1 fix. The campaigns endpoint was issuing N sequential calls to LeadByte (one per campaign). 28 campaigns × ~500ms = ~14s wall time. Replaced with 4 parallel /reports/campaign calls (one per window). Wall time = max single call.
  • Redis cache. 5-minute TTL on LeadByte responses; refuses to cache empty results so transient blips don't pin zeros.
  • Cache pre-warmer. Background worker runs every 90s, refreshes the cache before TTL expires. Users never see a cold-cache hit even after long idles.

Discovered along the way that the BullMQ workers weren't actually running in production — Railway's single-service deploy only runs the API. Fixed by spawning workers in the same Node process as the API. Side benefit: hourly LeadByte sync, ad-spend sync, bank-feed sync, and the daily overdue-invoice chase are all now firing for the first time.

  • /campaigns first call: 3-5 s → 1.9 s cold / 0.5 s warm
  • Idle > 60 s, next click: cold-miss every time → always warm
  • Last-sync timestamp on /integrations/leadbyte/status: null (never ran) → populated, refreshes every 2 min
Frontend: route-level code-splitting + preconnect + cache prefetch

Initial JS bundle dropped from ~500 KB to 71 KB gzip for first-time users. Each route loads its own chunk on demand. Recharts (95 KB gzip) only loads on chart pages.

  • <link rel=preconnect> to backend in index.html — saves ~150-300ms on first API call by warming TCP+TLS during HTML parse.
  • Login success fires parallel prefetch of dashboard endpoints — backend Redis is already warm by the time React renders the dashboard.
  • React Query staleTime aligned to backend cache TTL (5 min) so re-navigations don't force re-fetches.
  • Vercel region pinned to pdx1 to match Railway's US-West fleet — eliminates cross-continent FE→BE hop.
Lighthouse score: 100 / 100 / 100 on Accessibility, Best Practices, SEO
